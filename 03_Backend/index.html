REST-Grundprinzipien

Eindeutige Identifikation

Jede Ressource braucht zur Identifikation einen eindeutigen Schlüssel.

https://someurl/api/users/1
Verwendung von HTTP-Standardmethoden
GET https://someurl/api/users Alle User lesen
GET https://someurl/api/users/1 User 1 lesen
POST https://someurl/api/users Neuen User anlegen
PUT https://someurl/api/users/1 User 1 updaten
DELETE https://someurl/api/users/1 User 1 löschen

Verwendung von Hypermedia

Verwendung von HTML / HTTP als Basistechnologie

Beispielsweise können Query-String Parameter mitgegeben werden

https://someurl/api/users/?q=gerber

https://someurl/api/users/?limit=10

HTTP Standard https://tools.ietf.org/html/rfc2616

Unterschiedliche Repräsentationen von Ressourcen

Resultate beispielsweise als XML oder häufiger als JSON

Status Codes

Jeder HTTP Request an den Server wird mit einem StatusCode retourniert

  • 200 — Successful 2xx
  • 300 — Redirection 3xx
  • 400 — Client Error 4xx
  • 500 — Server Error 5xx

Wichtige Status Codes

Diese solltet ihr kennen

  • 200 — OK
  • 201 — Created
  • 204 — No Content
  • 301 — Moved Permanently
  • 302 — Found
  • 400 — Bad Request
  • 401 — Unauthorized
  • 403 — Forbidden
  • 404 — Not found
  • 409 — Conflict
  • 500 — Internal Server Error

Wann verwende ich GET?

Wenn kein method Attribut angegeben wird, wird standardmässig GET verwendet

  • Fügt Werte als Query-String (Name-Value-Pairs) an
  • Länge des Query-Strings ist beschränkt (2048 Zeichen)
  • Sollte nicht für sensitive Daten verwendet werden (Ist in der URL sichtbar)
  • Nützlich wenn eine Form Submission als Favorit gespeichert werden will (Z. Bsp. Google Suche)

Wann verwende ich POST?

Immer, wenn sensitive Daten übertragen werden müssen.

  • POST hat keine Grössenbeschränkungen und kann zum Senden grosser Datenmengen verwendet werden.
  • Kann nicht als Favorit (Bookmark) verwendet werden
